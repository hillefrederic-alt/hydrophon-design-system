# Labels und Helper Text

Beschriftungen und Hilfestellungen fuer Formularfelder.

## Labels

### Grundprinzipien
- Jedes Eingabefeld MUSS ein sichtbares Label haben
- Labels stehen UEBER dem Eingabefeld (nicht links, nicht als Placeholder)
- Labels beschreiben den erwarteten Inhalt praezise
- Kurz und praegnant (max 3-4 Woerter idealerweise)

### Technische Verknuepfung

```html
<!-- Pattern 1: for/id Verknuepfung -->
<label for="email">E-Mail-Adresse</label>
<input type="email" id="email" />

<!-- Pattern 2: Umschliessendes Label -->
<label>
  E-Mail-Adresse
  <input type="email" />
</label>
```

Empfehlung: for/id ist expliziter und ermoeglicht flexibleres Layout.

### Styling
- Schriftgroesse: 14px (font.size.sm)
- Schriftgewicht: 500 (medium)
- Farbe: neutral.700
- Abstand zum Input: 8px (spacing.2)

## Pflichtfeld-Kennzeichnung

### Visueller Indikator
- Rotes Asterisk (*) nach dem Label-Text
- Farbe: color.error (rot)
- Screenreader: aria-label="Pflichtfeld" auf dem Asterisk

### Legende
Am Formularanfang: "Felder mit * sind Pflichtfelder"

### HTML-Struktur
```html
<label for="name">
  Name
  <span class="required" aria-label="Pflichtfeld">*</span>
</label>
<input type="text" id="name" aria-required="true" required />
```

### Attribute
- `required`: Native HTML5 Validierung
- `aria-required="true"`: Screenreader-Ankuendigung

### Alternativer Ansatz
Wenn 80%+ der Felder Pflicht: "(optional)" bei optionalen Feldern statt Asterisk

## Helper Text

### Zweck
- Erklaert Eingabeformat (z.B. "TT.MM.JJJJ")
- Gibt Beispielwerte (z.B. "z.B. name@firma.de")
- Liefert zusaetzliche Kontextinformation
- Beschreibt Einschraenkungen (z.B. "Mindestens 8 Zeichen")

### Position
- UNTER dem Eingabefeld
- Abstand: 4px (spacing.1)
- Vor etwaiger Fehlermeldung

### Styling
- Schriftgroesse: 12px (font.size.xs)
- Farbe: neutral.600
- Zeilenhoehe: 1.4

### Placeholder vs Helper Text

| Placeholder | Helper Text |
|-------------|-------------|
| Verschwindet bei Eingabe | Bleibt sichtbar |
| Geringer Kontrast | Voller Kontrast |
| Nicht von Screenreadern als Label erkannt | Mit aria-describedby verknuepft |
| Fuer Beispielwerte wenn Platz fehlt | Fuer wichtige Informationen |

### Best Practice
```html
<label for="phone">Telefonnummer</label>
<input type="tel" id="phone" placeholder="+49 123 456789" aria-describedby="phone-helper" />
<span id="phone-helper" class="helper-text">
  Bitte mit Laendervorwahl
</span>
```

## Accessibility

### WCAG 3.3.2 Labels oder Anweisungen
- Alle Eingaben haben sichtbare Labels
- Labels sind programmatisch verknuepft (for/id oder umschliessend)
- Anweisungen werden VOR der Eingabe bereitgestellt

### aria-describedby
- Verknuepft Input mit Helper Text und/oder Error Message
- Mehrere IDs moeglich: `aria-describedby="helper error"`
- Screenreader liest nach Label und Input-Wert

### Fokus-Reihenfolge
1. Label (nicht fokussierbar, aber angekuendigt)
2. Input
3. Helper/Error Text (nicht fokussierbar, aber mit aria-describedby angekuendigt)

## Layout-Pattern

### Standard-Feld-Struktur
```
[Label] [Required Indicator]
[Input Field]
[Helper Text oder Error Message]
```

### Token-Referenzen
- input.label.marginBottom: 8px
- input.helper.marginTop: 4px
- form.spacing.fieldToField: 16px

## Anti-Patterns

### Vermeiden
- Placeholder als einziges Label
- Label rechts vom Input
- Floating Labels (Accessibility-Probleme)
- Label zu weit vom Input entfernt
- Hilfetext nur in Tooltip versteckt

```markdown
# Validierung und Fehlermeldungen

Patterns fuer Formular-Validierung mit klarem, hilfreichem Feedback.

## Validierungs-Strategie

### Progressive Validation (Empfohlen)
"Belohne frueh, bestrafe spaet"

1. **Initiale Eingabe**: Keine Validierung waehrend User tippt
2. **Blur (Feld verlassen)**: Erste Validierung
3. **Nach Fehler**: Echtzeit-Validierung bei jeder Aenderung
4. **Bei Korrektur**: Fehler verschwindet sofort

### React Hook Form Konfiguration
```typescript
const form = useForm({
  mode: 'onBlur',           // Initiale Validierung bei blur
  reValidateMode: 'onChange' // Nach Fehler: live re-validieren
});
```

### Warum dieses Pattern?
- Verhindert fruehe Fehler waehrend User noch tippt
- Gibt sofortiges Feedback sobald Korrektur erfolgt
- Reduziert Frustration und kognitive Last

## Error States

### Visueller Indikator
- Border: color.error (rot)
- Error-Icon: AlertCircle (Lucide) im Input rechts
- Error-Text unter dem Input

### Token-Werte
- input.field.border.error: {color.error}
- input.error.color: {color.error}
- input.error.fontSize: 12px

### WCAG-konforme Fehleranzeige
Fehler duerfen NICHT nur durch Farbe kommuniziert werden:
1. Rote Border (Farbe)
2. Error-Icon (Form)
3. Fehlertext (Text)
4. aria-invalid="true" (Programmatisch)

## Success States

### Visueller Indikator
- Border: color.success (gruen)
- Success-Icon: Check (Lucide) im Input rechts
- Kein zusaetzlicher Text noetig

### Verwendung
- Nach erfolgreicher Validierung (z.B. E-Mail-Format korrekt)
- Nicht bei jedem Feld noetig
- Besonders hilfreich bei komplexen Validierungen

## Fehlermeldungen

### Stil: Erklaerend-Hilfsbereit
Nicht nur "Ungueltig", sondern:
- Was ist falsch?
- Wie kann es korrigiert werden?
- Beispiel wenn hilfreich

### Beispiele

| Schlecht | Gut |
|----------|-----|
| "Ungueltig" | "Bitte gib eine gueltige E-Mail-Adresse ein (z.B. name@firma.de)" |
| "Fehler" | "Das Passwort muss mindestens 8 Zeichen lang sein" |
| "Falsches Format" | "Bitte gib das Datum im Format TT.MM.JJJJ ein" |
| "Pflichtfeld" | "Bitte gib deinen Namen ein" |

### Struktur einer guten Fehlermeldung
1. Was ist das Problem? (z.B. "Die E-Mail-Adresse ist ungueltig")
2. Wie beheben? (z.B. "Bitte pruefe das Format")
3. Beispiel (optional): (z.B. "z.B. name@firma.de")

### Ton
- Hoeflich, nicht anklagend
- Aktive Sprache ("Bitte gib..." statt "Es wurde nicht...")
- Konkret und hilfreich

## Error-Position

### Inline (Standard)
- Fehlermeldung direkt unter dem fehlerhaften Feld
- Sofort sichtbar im Kontext
- Empfohlen fuer einfache Formulare (1-5 Felder)

### Inline + Summary (Komplexe Formulare)
- Zusammenfassung oben im Formular
- Jeder Fehler verlinkt zum Feld
- Focus springt zur Summary bei Submit
- Empfohlen fuer 5+ Felder oder Multi-Step

### Summary-Pattern
```html
<div class="error-summary" role="alert" tabindex="-1">
  <h2>Bitte korrigiere folgende Fehler:</h2>
  <ul>
    <li><a href="#email">E-Mail-Adresse ist ungueltig</a></li>
    <li><a href="#password">Passwort zu kurz</a></li>
  </ul>
</div>
```

## Accessibility

### WCAG 3.3.1 Fehlererkennung
- Fehler werden automatisch erkannt und beschrieben
- Fehlerhafte Felder werden identifiziert

### WCAG 3.3.3 Fehlervorschlaege
- Wenn bekannt, wird Korrekturvorschlag gegeben
- Beispielwerte wo hilfreich

### aria-invalid
```html
<input
  type="email"
  aria-invalid="true"
  aria-describedby="email-error"
/>
<span id="email-error" role="alert">
  Bitte gib eine gueltige E-Mail-Adresse ein
</span>
```

### role="alert"
- Error-Messages mit role="alert"
- Wird sofort von Screenreadern angekuendigt
- Impliziert aria-live="assertive"

### Focus-Management bei Submit
1. Bei Fehlern: Focus auf erstes fehlerhaftes Feld ODER Error-Summary
2. Error-Summary mit tabindex="-1" fuer programmatischen Focus
3. Links in Summary fuehren zu den Feldern

## Formular-Level-Validierung

### Bei Submit
- Alle Felder validieren
- Alle Fehler gleichzeitig anzeigen
- Focus auf ersten Fehler oder Summary

### Submit-Button-Zustand
- NICHT deaktivieren basierend auf Validierung
- Immer klickbar lassen
- Validierung bei Klick ausfuehren
- Deaktivieren nur waehrend Submission (loading)

## Vollstaendiges Integrations-Beispiel

Das folgende Beispiel zeigt alle Teile zusammen: Token-Referenzen, React Hook Form, Zod Schema, ARIA-Attribute und visuelles Styling.

### Schema (Zod)
```typescript
import { z } from 'zod';

const contactSchema = z.object({
  name: z
    .string()
    .min(1, 'Bitte gib deinen Namen ein')
    .min(2, 'Der Name muss mindestens 2 Zeichen haben'),
  email: z
    .string()
    .min(1, 'E-Mail-Adresse ist erforderlich')
    .email('Bitte gib eine gueltige E-Mail-Adresse ein (z.B. name@firma.de)'),
  message: z
    .string()
    .min(1, 'Bitte gib eine Nachricht ein')
    .min(10, 'Die Nachricht muss mindestens 10 Zeichen haben'),
  terms: z
    .boolean()
    .refine(val => val === true, 'Bitte akzeptiere die Nutzungsbedingungen')
});
```

### React Component
```tsx
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';

function ContactForm() {
  const {
    register,
    handleSubmit,
    formState: { errors, isSubmitting }
  } = useForm({
    resolver: zodResolver(contactSchema),
    mode: 'onBlur',
    reValidateMode: 'onChange'
  });

  const onSubmit = async (data) => {
    // Submit logic
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)} noValidate>
      {/* Name Field */}
      <div className="form-field">
        <label htmlFor="name">
          Name
          <span className="required" aria-label="Pflichtfeld">*</span>
        </label>
        <input
          id="name"
          type="text"
          aria-required="true"
          aria-invalid={!!errors.name}
          aria-describedby={errors.name ? 'name-error' : undefined}
          {...register('name')}
        />
        {errors.name && (
          <span id="name-error" className="error-message" role="alert">
            {errors.name.message}
          </span>
        )}
      </div>

      {/* Email Field */}
      <div className="form-field">
        <label htmlFor="email">
          E-Mail-Adresse
          <span className="required" aria-label="Pflichtfeld">*</span>
        </label>
        <input
          id="email"
          type="email"
          aria-required="true"
          aria-invalid={!!errors.email}
          aria-describedby={errors.email ? 'email-error' : 'email-helper'}
          {...register('email')}
        />
        <span id="email-helper" className="helper-text">
          Wir verwenden deine E-Mail nur zur Kontaktaufnahme
        </span>
        {errors.email && (
          <span id="email-error" className="error-message" role="alert">
            {errors.email.message}
          </span>
        )}
      </div>

      {/* Terms Checkbox */}
      <div className="form-field">
        <label className="checkbox-label">
          <input
            type="checkbox"
            aria-required="true"
            aria-invalid={!!errors.terms}
            {...register('terms')}
          />
          <span className="checkbox-custom"></span>
          Ich akzeptiere die Nutzungsbedingungen
          <span className="required" aria-label="Pflichtfeld">*</span>
        </label>
        {errors.terms && (
          <span className="error-message" role="alert">
            {errors.terms.message}
          </span>
        )}
      </div>

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Wird gesendet...' : 'Absenden'}
      </button>
    </form>
  );
}
```

Vollstaendiges Formularsystem fuer konsistente Dateneingabe mit WCAG 2.1 AA Compliance.

## Uebersicht

Dieses Kapitel dokumentiert alle Form-Komponenten des Hydrophon Design Systems.

### Schnellzugriff

|------------|--------------|----------|

## Design-Prinzipien

### Konsistenz mit Button-System
- Gleiche Hoehen: Small (32px), Medium (40px), Large (48px)
- Gleiche Border-Radius: 4px
- Gleiche Focus-Indikatoren: 2px Outline

### Accessibility First
- Native HTML-Elemente wo moeglich
- Vollstaendige Tastaturnavigation
- Screenreader-kompatibel
- WCAG 2.1 AA konform

### Outline-Stil
- Weisser Hintergrund
- Sichtbare Border
- Klarer Focus-State

## Groessen

| Groesse | Hoehe | Verwendung |
|---------|-------|------------|
| Small | 32px | Kompakte UIs, nur Desktop |
| Medium | 40px | Standard, meiste Formulare |

- Default
- Hover
- Focus
- Error
- Success (wo sinnvoll)
