<task type="auto">
  <name>Task 2: Breadcrumb-Dokumentation erstellen</name>
  <files>design-system/docs/navigation/breadcrumb.md</files>
  <action>
Erstelle breadcrumb.md auf Deutsch mit mindestens 300 Zeilen:

**Struktur:**

## Übersicht
- Zweck: Hierarchische Position anzeigen
- Wann verwenden: Seiten mit 3+ Tiefe, externe Einstiege (Suche, Links)
- B2B-Kontext: Produkt-Kategorien, Unternehmensstruktur

## Anatomie
- nav-Element mit aria-label="Breadcrumbs"
- ol-Element (ordered list für Reihenfolge)
- li-Elemente mit Links
- Letztes Element: span (kein Link) mit aria-current="page"

## Separator
### Chevron (›) Empfohlen
- Modern, richtungsweisend
- CSS pseudo-element ::after
- Automatisch vor Screen Readern versteckt

### Alternative Separatoren
- Slash (/) - klassisch
- Pfeil (→) - stark richtungsweisend
- Token-basiert: leicht global änderbar

## Mobile-Verhalten
- Option 1: Vollständige Breadcrumbs mit Wrap
- Option 2: Nur Home + Current (mittlere Elemente ausblenden)
- Media Query Beispiel
- Empfehlung: Vollständig bei < 4 Elementen, sonst gekürzt

## Tokens
- Token-Tabelle mit Namen, Werten, Beschreibungen
- CSS-Variable-Mapping

## Code-Beispiele
### HTML
- Vollständige semantische Struktur
- aria-label auf nav
- aria-current="page" auf aktuellem Element
- Kein Link auf aktuellem Element

### CSS
- Flexbox-Layout mit gap
- ::after für Separator
- Current-Styling
- Mobile-Anpassung

## Barrierefreiheit
- WCAG 2.4.8: Location (Breadcrumbs erfüllen dieses Kriterium)
- aria-current="page" (muss vorhanden sein)
- Separator: Pseudo-Element = automatisch versteckt
- Kontrast: 4.5:1 für Links
- Focus-Indicator wie alle Links

## Best Practices / Nicht verwenden
- Best Practices: Immer mit aria-current, Separator als ::after, ol statt ul
- Nicht verwenden: Separator als echtes HTML-Element, Link auf aktuelle Seite, fehlende aria-label

## Integration mit Header
- Breadcrumbs unter Header positionieren
- Konsistente Container-Breite
- Visueller Bezug zur Navigation

Format: Markdown mit deutschen Überschriften, Code-Blöcken, Token-Tabellen.

  <files>design-system/docs/navigation/footer.md</files>
  <action>
Erstelle footer.md auf Deutsch mit mindestens 250 Zeilen:

**Struktur:**

## Übersicht
- Zweck: Sekundäre Navigation, Kontakt, rechtliche Pflichtangaben
- B2B-Kontext: Vertrauensbildung, Professionalität

## Anatomie
- footer-Element mit role="contentinfo"
- Inhaltscontainer
- Link-Gruppen (3-4 Spalten)
- Legal-Bereich (Copyright, Impressum, Datenschutz)

## Layout
### Desktop (>= 768px)
- CSS Grid: 3-4 Spalten
- Logo optional links
- Link-Gruppen in Spalten
- Legal-Bereich volle Breite unten

### Mobile (< 768px)
- Einspaltig
- Akkordeon oder gestackt
- Legal-Bereich unten

## Link-Gruppen
### Typische Gruppen
- Produkte (Links zu Produktkategorien)
- Unternehmen (Über uns, Karriere, etc.)
- Service (Support, Downloads, FAQ)
- Kontakt (Adresse, Telefon, E-Mail)

### Styling
- Überschrift: Uppercase, semibold, kleiner
- Links: Einfarbig, Hover-State
- Touch-Target: line-height 2 für Mobile

## Legal-Bereich
- Copyright-Text
- Pflicht-Links: Impressum, Datenschutz, AGB
- Separator: | oder inline-flex mit gap
- Kleinere Schrift, dezenter

## Tokens
- Token-Tabelle mit Namen, Werten, Beschreibungen
- CSS-Variable-Mapping

## Code-Beispiele
### HTML
- Semantische footer-Struktur
- nav-Element für Link-Gruppen (aria-label für jede Gruppe)
- address-Element für Kontakt (optional)

### CSS
- Grid-Layout für Spalten
- Responsive Anpassung
- Focus-States

## Barrierefreiheit
- Landmark: role="contentinfo" (automatisch durch footer)
- Mehrere nav-Elemente: Jeweils mit aria-label
- Skip-Link: Footer sollte übersprungen werden können
- Kontrast auf farbigem Hintergrund beachten

## Best Practices / Nicht verwenden
- Best Practices: Klare Gruppierung, max 4 Spalten, Legal immer unten
- Nicht verwenden: Zu viele Links ohne Gruppierung, keine Landmark-Region

Format: Markdown mit deutschen Überschriften, Code-Blöcken, Token-Tabellen.
